Notes / open questions

  • Oklahoma filters are live — every search re-queries OCC's real API. Texas filters are client-side against a small pre-fetched sample — the real ingestion job will need scheduled bulk downloads plus a proper indexed database to support this kind of filtering server-side once it's not just a preview.
  • OCC's COMP_WELLS table is completion reports, not every well ever permitted — a well with a permit but no completion filed yet won't show up there (it'll be in ITD_WELLS as an application instead). Worth confirming with OCC docs whether there's a gap between "permit approved" and "completion filed" where a spudded, actively-drilling well is invisible to both layers.
  • County identifiers are inconsistent across and even within sources (OCC wells: "GRADY"; OCC permits: same; TX permits: numeric code only, e.g. "003"; TX wells: plain name). This viewer maps TX's numeric codes to names using RRC's own published table so both TX tables can share one county filter — confirms the spec's plan to standardize on FIPS codes in the normalization layer rather than trusting raw strings.
